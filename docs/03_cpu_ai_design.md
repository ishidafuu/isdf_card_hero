# CPU AI 設計メモ

## 目的

作業スレッドでCPU AIを実装するための設計メモ。

現行プロトタイプのCPUは、固定優先順位に従って最低限ターンを進めるためのもの。
次の段階では、強いAIを作るより先に、検証相手として破綻しにくく、行動理由を追いやすいAIを作る。

重視すること:

- 既存の戦闘ルール関数を使い、AI側でルールを再実装しない。
- 同じ盤面なら同じ手を選ぶ決定的なAIにする。
- なぜその手を選んだかをログやテストで確認できるようにする。
- ルール追加に耐えられるよう、優先順位ベタ書きではなく「合法手列挙 + 評価関数」に寄せる。
- ただし、最初からminimaxやMCTSのような重い探索にはしない。

## 現状

現行仕様では、CPUは検証用の単純AI。

優先順位:

1. 勝てる攻撃があればマスターを攻撃する。
2. 倒せる敵モンスターがいれば攻撃する。
3. 召喚できるなら前衛、次に後衛へ召喚する。
4. 攻撃できる対象があれば攻撃する。
5. ためることができるモンスターがいればためる。
6. 何もできなければターン終了する。

現行仕様上、CPUは次を使う。

- モンスター攻撃
- モンスター召喚
- マスターアタック
- ためる
- 自動レベルアップ
- 自動手札廃棄

現行仕様上、CPUは次を使わない。

- マスターHPドロー
- マジックカード
- ウェイクアップ
- シールド
- 任意移動

このため、最初のAI改善では「使わない」と仕様化されている行動を勝手に解禁しない。
マジックや移動を使うAIは、仕様書側のCPU節を更新してから実装する。

## 推奨する全体設計

CPU AIは次の構成にする。

1. `listCpuDecisions(state)` で、現在のCPUが選べる合法手を列挙する。
2. `applyCpuDecision(state, decision)` で、既存ルール関数を使って仮実行する。
3. `evaluateState(state)` で、CPU視点の盤面点を出す。
4. `chooseCpuDecision(state)` で、合法手の中から最良手を選ぶ。
5. `runCpuStep` は、最終的に `chooseCpuDecision` の結果を1手だけ適用する。

最初は1手評価で十分。
ただし、将来 `パワーアップ -> 攻撃` や `移動 -> 攻撃` を扱えるよう、設計上は浅い探索へ拡張しやすくしておく。

## 競合しにくい実装方針

別スレッドで戦闘実装が進んでいるため、CPU AI本体は新規ファイルに分ける。

推奨ファイル:

- `src/game/cpuAi.ts`

既存ファイルへの変更は最小限にする。

- `src/game/rules.ts`
  - `runCpuStep` から `chooseCpuDecision` / `applyCpuDecision` を呼ぶ接続だけにする。
  - 既存のルール本体はできるだけ触らない。

ただし、既存の `rules.ts` 内にprivate helperが多い場合は、AI側でprivate関数に依存しない。
公開済みの関数だけで合法手を作る。

主に使う公開関数:

- `getCommandTargets`
- `getMonsterCommands`
- `getMasterActionTargets`
- `getMagicTargets`
- `getMovableTargets`
- `canSummonTo`
- `canFocusMonster`
- `attackWithCommand`
- `summonMonster`
- `useMasterAction`
- `focusMonster`
- `playMagic`
- `moveMonster`
- `endTurn`

カード情報は `getCardDef` / `getMonsterDef` を使う。

## データ型案

AIの判断は、実行可能な1手として表現する。

```ts
export type CpuDecision =
  | {
      type: "attack";
      action: CommandAction;
      reason: string;
      score: number;
    }
  | {
      type: "master_action";
      actionId: MasterActionId;
      target: Target;
      reason: string;
      score: number;
    }
  | {
      type: "summon";
      handInstanceId: string;
      slotKey: SlotKey;
      reason: string;
      score: number;
    }
  | {
      type: "magic";
      action: MagicAction;
      reason: string;
      score: number;
    }
  | {
      type: "move";
      fromSlotKey: SlotKey;
      toSlotKey: SlotKey;
      reason: string;
      score: number;
    }
  | {
      type: "focus";
      slotKey: SlotKey;
      reason: string;
      score: number;
    }
  | {
      type: "end_turn";
      reason: string;
      score: number;
    };
```

v1.1では `magic` と `move` は列挙しない。
v1.2以降で仕様を更新してから有効化する。

## v1.1: 現行仕様内の改善

最初に実装する範囲。

対象行動:

- モンスター攻撃
- マスターアタック
- モンスター召喚
- ためる
- ターン終了

実装しない行動:

- マジック
- ウェイクアップ
- シールド
- 任意移動
- マスターHPドロー

### 合法手列挙

攻撃:

- CPU自陣の登場済みモンスターをフィールド順に見る。
- 各モンスターのコマンドを列挙する。
- `getCommandTargets` で合法対象を列挙する。
- 味方モンスターへの攻撃は、v1.1では候補から除外する。

マスターアタック:

- `getMasterActionTargets(state, "master_attack")` で対象を列挙する。
- 対象は敵前列の登場済みモンスターのみ。

召喚:

- CPU手札のモンスターカードを列挙する。
- CPU自陣の空き枠を列挙する。
- `canSummonTo` で合法性を確認する。
- カード役割と配置で評価する。

ためる:

- CPU自陣の登場済みモンスターを列挙する。
- `canFocusMonster` がtrueのものだけ候補にする。

ターン終了:

- 常に候補に含める。
- 他の行動が十分に有利でない場合の逃げ道にする。

### 召喚評価

現行CPUの弱点は、空き枠と手札を先頭から選ぶため、後衛カードを前に出しやすいこと。
v1.1では召喚評価を最優先で改善する。

基本方針:

- 前衛カードは前列へ置く。
- 後衛カードは後列へ置く。
- 後衛カードは、同じレーンの前列が埋まっている後列を高く評価する。
- 後衛カードを、同じレーンの前列が空いている後列へ置く評価は下げる。
  - 次の自ターン開始時に前列へ自動移動し、後衛らしい働きがしにくいため。
- 前衛カードを後列に置く評価は下げる。
- モーガンは `any_target` を持つため、他の後衛より前列配置の減点を軽くしてよい。

召喚スコア例:

```text
前衛カードを前列へ召喚: +45
前衛カードを後列へ召喚: +15
後衛カードを後列へ召喚: +40
後衛カードを、同レーン前列が埋まっている後列へ召喚: +20
後衛カードを、同レーン前列が空いている後列へ召喚: -20
後衛カードを前列へ召喚: -10
盤面が空で前衛カードを先に出せる: +15
```

### 攻撃評価

攻撃は次を重視する。

- 相手マスターを倒せる攻撃は最優先。
- 敵モンスターを倒せる攻撃は高評価。
- レベルアップ余地のある自軍モンスターで撃破できるなら追加評価。
- 非致死のマスター攻撃は評価するが、相手にストーンを渡すため過大評価しない。
- 0ダメージ攻撃は基本的に選ばない。
- 味方攻撃はv1.1では選ばない。

攻撃スコア例:

```text
相手マスター撃破: +1_000_000
敵モンスター撃破: +300 + 対象価値
レベルアップ可能な撃破: +80 * 上げられる推定レベル数
相手マスターへの実ダメージ: +70 * ダメージ
敵モンスターへの非致死ダメージ: +25 * ダメージ
攻撃後に自爆反動で自分が倒れる: -120
0ダメージ: -100
```

ボムゾウの反動は必ず評価に入れる。
反動で自分が倒れる場合、撃破やリーサル以外では大きく減点する。

### マスターアタック評価

マスターアタックは3ストーン消費で2P。
雑に使うとストーン効率が悪い。

方針:

- 敵モンスターを倒せる場合は高評価。
- 倒せない場合は低評価。
- ストーンが潤沢で、敵主力を削れる場合だけ中程度に評価する。

スコア例:

```text
敵モンスター撃破: +220 + 対象価値
非致死ダメージ: +10 * ダメージ - 3ストーン消費ペナルティ
```

### ためる評価

ためるは、攻撃できない場合や攻撃価値が低い場合の選択肢。

方針:

- すでに攻撃価値の高い手があるなら、ためるは選びにくくする。
- 次の攻撃でマスターシールドを突破できる可能性があるモンスターは高めに評価する。
- 後衛アタッカーのためるは、射程攻撃の価値が高いので少し評価する。

スコア例:

```text
通常のためる: +18
次の攻撃で3P以上になれる: +20
すでに前線で倒されやすい低HPモンスター: -10
```

## 評価関数

盤面評価はCPU視点で数値化する。

初期案:

```text
勝利: +1_000_000
敗北: -1_000_000
マスターHP差: (cpuHp - playerHp) * 80
ストーン差: (cpuStones - playerStones) * 6
手札差: (cpuHand - playerHand) * 3
山札差: (cpuDeck - playerDeck) * 1
```

モンスター価値:

```text
基礎価値 = 30
レベル価値 = level * 35
HP価値 = hp * 6
投資ストーン価値 = investedStones * 8
行動可能ボーナス = actionCount < actionLimit ? 12 : 0
気合いボーナス = focused ? 18 : 0
パワーアップボーナス = powerUp ? 16 : 0
シールドボーナス = shielded ? 12 : 0
準備中係数 = status === "prepared" ? 0.6 : 1.0
```

配置価値:

```text
前衛カードが前列: +12
前衛カードが後列: -8
後衛カードが後列: +16
後衛カードが前列: -12
後衛カードが後列かつ同レーン前列が埋まっている: +10
後衛カードが後列かつ同レーン前列が空: -12
```

最終的に、自軍モンスター価値を加算し、敵軍モンスター価値を減算する。

## 探索方針

### v1.1

1手評価だけでよい。

```text
best = max(decisionScore + evaluateState(after) - evaluateState(before))
```

ただし、同点時の順序は固定する。
ランダム選択はしない。

同点タイブレーク例:

1. 勝利
2. 敵撃破
3. 召喚
4. 非致死攻撃
5. ためる
6. ターン終了
7. フィールド表示順
8. 手札順

### v1.2以降

浅いビーム探索を導入する。

推奨値:

```text
depth: 3から5
beamWidth: 6から10
```

同一ターン内の連続行動だけを見る。
相手ターンまで読むminimaxはまだ不要。

探索を入れる理由:

- パワーアップから攻撃する。
- ウェイクアップから攻撃する。
- マスターアタックで削ってからモンスターで倒す。
- 召喚より先に攻撃した方がレベルアップ後のストーン消費に合う。

## v1.2: マジック解禁

仕様書のCPU節を更新してから実装する。

対象:

- サンダー
- ヒーリング
- パワーアップ

### サンダー

使う条件:

- 相手マスターを倒せる。
- 敵モンスターを倒せる。
- 次の攻撃と組み合わせて重要な敵を倒せる。

使わない条件:

- 非致死で相手マスターに1ダメージだけ与える目的。
- ストーンを使い切って召喚やレベルアップができなくなる。

### ヒーリング

使う条件:

- Lv2以上、または高価値モンスターが次に倒されそう。
- 回復量が2点近く無駄なく入る。

使わない条件:

- 1点だけの回復。
- 低価値モンスターの延命。
- 攻撃や召喚の方が明確に強い。

### パワーアップ

使う条件:

- 同ターン中に攻撃できるモンスターへ使う。
- 使用後の攻撃でマスターシールドを突破する。
- 使用後の攻撃で敵モンスターを倒す。

使わない条件:

- 攻撃予定のないモンスターへの先置き。
- すでにパワーアップ済みのモンスター。

## v1.3: ウェイクアップ、シールド、移動

仕様書のCPU節を更新してから実装する。

### ウェイクアップ

使う条件:

- 自分の準備中モンスターを起こして、そのまま有効攻撃できる。
- 相手の準備中モンスターを起こして、そのまま倒せる。

注意:

- ウェイクアップでは自動前進が発生しない。
- 起こした自軍モンスターが同ターンに行動できることを評価する。

### シールド

使う条件:

- 高価値モンスターが次ターン倒される可能性を下げる。
- レベルアップ済みモンスターや後衛主力を守る。

初期段階では、相手の次手を完全には読まず、簡易危険度でよい。

### 任意移動

使う条件:

- 後衛カードを後列へ戻せる。
- 前衛カードを前列へ出せる。
- 射程が通る位置へ移動できる。
- 空き前列を埋めてマスター周辺を守れる。

注意:

- 移動は行動を消費するため、同ターンに攻撃できる手より優先しすぎない。

## v2: 相手ターンを軽く読む

v1でCPUが破綻せず動くようになってから検討する。

最初から完全なminimaxは不要。
相手の次ターンについて、次だけを推定する。

- 相手が次にCPUマスターへ出せる最大実ダメージ
- 自軍モンスターのうち、次に倒されそうなもの
- 相手が次にレベルアップできそうな撃破

危険度評価例:

```text
次ターンCPUマスターが倒される: -800_000
次ターンCPUマスターが3点以上受ける: -180
Lv2以上の自軍モンスターが倒されそう: -160
相手が撃破でレベルアップしそう: -120
```

## ログとデバッグ

AI選択理由は、最初から短い文字列で持たせる。

例:

- `相手マスターを倒せるため攻撃`
- `敵モンスターを撃破できるため攻撃`
- `後衛カードを守られた後列へ召喚`
- `有効攻撃がないためためる`
- `有効な行動がないためターン終了`

ログへ毎回出すかはUIの見やすさ次第。
少なくともテストでは `reason` を確認できる形にする。

## テスト方針

AIはスナップショット的な総合テストだけにしない。
小さい盤面を作って、選択されるべき1手を検証する。

v1.1で追加したいテスト:

- 相手マスターを倒せる攻撃があるなら、それを選ぶ。
- 敵モンスターを倒せる攻撃があるなら、非致死マスター攻撃より優先する。
- 味方モンスターへの攻撃を選ばない。
- 0ダメージ攻撃を選ばない。
- マスターアタックは、撃破できる対象がいるときだけ高く評価される。
- 前衛カードは前列へ召喚する。
- 後衛カードは、同レーン前列が埋まっている後列へ召喚する。
- 有効な攻撃がないとき、行動可能なモンスターはためる。
- 20から100 seed程度のCPU自動ターンで、例外や無限行動が起きない。

v1.2以降で追加したいテスト:

- サンダーでリーサルできるなら使う。
- サンダーで敵モンスターを倒せるなら候補にする。
- パワーアップは同ターン攻撃とセットで使う。
- ヒーリングは高価値モンスターにだけ使う。
- ウェイクアップ後に攻撃できる場合だけ使う。
- シールドは高価値モンスターを守る場合だけ使う。
- 移動は攻撃機会を捨てすぎない。

## 実装手順案

1. `src/game/cpuAi.ts` を追加する。
2. `CpuDecision` 型を定義する。
3. v1.1対象の合法手列挙を作る。
4. `applyCpuDecision` を作り、既存ルール関数だけで実行する。
5. `evaluateState` を作る。
6. `chooseCpuDecision` を1手評価で作る。
7. `rules.ts` の `runCpuStep` から接続する。
8. v1.1の小さい盤面テストを追加する。
9. `npm test` と `npm run build` を通す。
10. 必要ならCPU理由ログを追加する。

## 非目標

v1では次をしない。

- MCTS
- 深いminimax
- ランダムな手選択
- デッキ読み
- 長期的な手札温存戦略
- 対人レベルの強さ調整
- 公式CPUの再現

今必要なのは、強すぎるAIではなく、戦闘プロトタイプを何戦も回せる検証相手。
そのため、説明可能で壊れにくいヒューリスティックAIを優先する。
