# プロジェクト方針

## 目的

カードヒーロー風の戦闘を出発点に、まずはカードヒーローとしての通常戦闘の完成度を
上げる。

ただし、最初から完成形を作らない。まずは小さい戦闘プロトタイプを作り、
位置取り、育成、リソース管理が面白いかを確認する。

## 現時点の前提

- まずはカードヒーローとして遊べる通常戦闘プロトタイプを優先する。
- 参照する中心要素は、通常バトルの盤面、前衛/後衛、召喚準備、ストーン、マスターHP、レベルアップ。
- 最初の目標はローグライトではなく、1戦の戦闘を成立させること。
- 公式ルールは資料で確認し、採用、簡略化、未採用を分けてから実装する。
- プロトタイプでも、採用したルールは曖昧なまま実装しない。
- Phase 1の実装技術は Vite + React + TypeScript とし、ルール本体はReactに依存しない純TypeScriptで持つ。

## 面白さの仮説

このプロジェクトで最初に検証したい面白さは、カード枚数の多さではなく、
次の判断が毎ターン発生すること。

- どのマスにユニットを置くか
- どのユニットで攻撃するか
- どのユニットに撃破経験を取らせるか
- ストーンを召喚、攻撃、回復、温存のどれに使うか
- マスターを守るか、相手マスターを詰めに行くか

## 当面の非目標

- オンライン対戦
- 大量のカード実装
- ストーリー
- 豪華な演出
- 複雑なマップイベント
- 肉、変異、ローグライト進行
- 既存作品の公式AI完全再現

## 段階的なロードマップ

### Phase 1: 最小戦闘

1戦だけ遊べる状態を作る。
この段階から、カードヒーローらしさに直結する召喚準備、ストーン、前衛/後衛、
レベルアップは最小形で入れる。

確認したいこと:

- 盤面にユニットを出して攻撃するだけで、判断が発生するか
- マスターHPを削る勝利条件がわかりやすいか
- 操作の流れが重くないか

### Phase 2: カードヒーローらしさの追加

攻撃範囲、カード効果、マジック、レベルアップ選択を広げる。

確認したいこと:

- 位置取りが戦術になっているか
- 撃破を誰に取らせるかが悩ましいか
- ストーンが単なるコストではなく、テンポ管理になっているか

### Phase 3: CPU対戦

カード追加より先にCPUを作る。最初は仕様化した優先順位に従う単純AIにする。

確認したいこと:

- 人間同士でなくても戦闘検証を回せるか
- CPU相手に20戦ほど回して破綻しないか

### Phase 4: カードセット検証

通常カードの効果、性格、射程、ランダム解決、CPU利用を検証する。
未対応効果が表示やログに混ざらない状態を維持する。

### Phase 5: プレイアビリティ改善

カード情報、ログ、行動理由、デッキ/手札/捨て札確認を強化する。
手動プレイ時に、なぜ行動できないか、なぜ効果が発火したかを追える状態にする。

### Phase 6: カードヒーロー完成度向上

通常戦として不足しているマスター差分、対戦設定、デッキ構築、スペシャルカード方針、
公式挙動との差分を詰める。

### Phase 7: 対戦モード整備

プレイヤー対CPU、CPU対CPU、固定seed検証、デッキ選択を自然につなげる。

### Phase 8: 拡張要素の再判断

カードヒーロー通常戦が十分に整ってから、ローグライト化や独自要素を再判断する。

## 最初の到達点

最初の実装ゴールは、次の条件を満たす戦闘プロトタイプ。

- プレイヤーとCPU、またはローカル2人操作で1戦が完了する
- 前後 x 左中右の2 x 3盤面で、前列中央にマスター、後列中央に進入不可マスがある
- 前列左右、後列左右の4枠へユニットを召喚できる
- 召喚直後は準備中になり、次の自ターンに登場する
- ユニットとマスターにHPがある
- 攻撃でHPが減る
- マスターには2パワーシールドがある
- ストーンで召喚、マジック、レベルアップを行う
- 撃破したモンスターがレベルアップできる
- 相手マスターのHPを0にすると勝利する
- ランダム30枚デッキだけで数戦試せる

## 後で決めること

- Phase 1後に盤面サイズを変えるか
- Phase 1後に公式ルールをどこまで追加採用するか
- ランダムデッキ検証後に正式なデッキ構築をいつ入れるか
- ローグライト化する前に、どの程度CPUを賢くするか
