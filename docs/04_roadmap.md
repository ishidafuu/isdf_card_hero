# 今後のロードマップ

## 目的

このロードマップは、現在の戦闘プロトタイプを土台に、検証可能な順番で機能を積み上げるための作業順を定義する。

最優先は、カード追加やローグライト化ではなく、1戦の戦闘が安定して何度も検証できる状態にすること。

## 現在地

完了済み:

- Vite + React + TypeScript の戦闘プロトタイプ基盤
- 2 x 3盤面、マスター位置、進入不可マス
- 固定30枚デッキと同名3枚制限
- 召喚、準備中、登場、自動前進
- ストーン、マスターHPドロー、山札切れペナルティ
- モンスター攻撃、マスターアタック、マジック、ためる
- 味方攻撃と味方撃破ペナルティ
- 撃破時レベルアップ、投資ストーン返却
- CPU AI v1.1
  - モンスター攻撃
  - マスターアタック
  - モンスター召喚
  - ためる
  - ターン終了

## 優先順位

### Phase 1: 戦闘プロトタイプの安定化

目的:

現在の1戦プロトタイプを、手動プレイとCPU相手の反復検証に耐える状態にする。

実装対象:

- CPU選択理由のログ表示
- ターン中に何が起きたかを追いやすくするログ分類と表示改善
- 操作可能対象、攻撃可能対象、選択中カードの視認性改善
- 勝敗後のリスタート導線
- 主要ルールの回帰テスト追加

完了条件:

- 20戦程度の手動プレイで、進行不能、例外、意図しない待ち状態が出ない。
- CPUターン中に、召喚、攻撃、ためる、ターン終了の理由をログから追える。
- 主要アクションの対象選択で、プレイヤーが次に押せる場所を迷わない。

### Phase 2: CPU AI v1.2 マジック解禁

目的:

CPUがマジックカードを使えるようにし、現行カードセットの検証精度を上げる。

実装対象:

- `thunder` のAI候補化
- `healing` のAI候補化
- `power_up` のAI候補化
- マジックを含む1手評価
- 必要なら同一ターン内だけの浅い探索

着手前に確定する仕様:

- CPUが非致死サンダーを相手マスターへ使ってよい条件
- CPUがヒーリングを温存するか、使える時点で使うか
- パワーアップを単独手として扱うか、攻撃とセットで評価するか

完了条件:

- サンダーでリーサルできる場合は使う。
- サンダーで敵モンスターを倒せる場合は候補にする。
- ヒーリングは高価値モンスターへ無駄なく入る場合だけ使う。
- パワーアップは同ターン攻撃で利益がある場合だけ使う。

### Phase 3: CPU AI v1.3 マスター特技と移動

目的:

CPUがウェイクアップ、シールド、任意移動を使えるようにし、位置取りと防御判断を検証できる状態にする。

実装対象:

- ウェイクアップ候補の列挙
- シールド候補の列挙
- 任意移動候補の列挙
- CPU視点の簡易危険度評価
- 移動後攻撃、ウェイクアップ後攻撃を扱える浅い探索

着手前に確定する仕様:

- ウェイクアップで敵準備中モンスターを起こす戦術をCPUに許可するか
- シールドをモンスター防御専用にするか、将来マスター防御へ拡張するか
- 任意移動で味方同士の入れ替えをCPUに許可する優先条件

完了条件:

- CPUが有効攻撃につながるウェイクアップを選べる。
- CPUが高価値モンスターを守るシールドを選べる。
- CPUが後衛を後列へ戻す、前衛を前列へ出す移動を選べる。
- 移動だけでターンを浪費し続けない。

### Phase 4: プレイアビリティ改善

目的:

ルールは動いているが遊びづらい状態を減らし、繰り返し検証しやすくする。

実装対象:

- 行動履歴のフィルタ、色分け、重要ログ強調
- CPUターン速度設定
- 盤面上のダメージ、回復、撃破、レベルアップ表示
- カード詳細パネル
- 山札、捨て札、手札の確認UI

完了条件:

- ログを追わなくても、直近の行動結果が盤面上で理解できる。
- カード効果と現在状態を画面上で確認できる。
- CPUターン速度を検証用に調整できる。

### Phase 5: カードセット拡張

目的:

少数カードの挙動確認から、戦術の幅を検証する段階へ進む。

実装対象:

- 追加モンスターカード
- 追加マジックカード
- カード定義の整理
- カード効果ごとのテストテンプレート

着手前に確定する仕様:

- 公式寄りのカードを増やすか、独自カードを混ぜるか
- 追加カードを固定デッキへ入れるか、別デッキとして扱うか
- スーパーカード化を採用するか

完了条件:

- 固定デッキ以外に、検証用デッキを複数選べる。
- カード追加時に最低限のルールテストを追加できる。
- 追加カードが既存戦闘ルールを壊していない。

### Phase 6: 戦闘後報酬と肉システム

目的:

カードヒーロー風戦闘から、このプロジェクト固有の面白さへ進む。

実装対象:

- 戦闘結果画面
- 撃破報酬
- 肉の獲得
- 肉による変異
- 変異候補の提示

着手前に確定する仕様:

- 肉はモンスター単位で持つか、プレイヤー全体の報酬にするか
- 変異はカードを書き換えるか、ラン中の個体だけを書き換えるか
- 死亡、変異、成長をどの程度永続化するか

完了条件:

- 1戦後に報酬選択が発生する。
- 肉を選ぶことで、次の戦闘に明確な変化が出る。
- 変異結果がプレイヤーに理解できる。

### Phase 7: ローグライト最小ループ

目的:

単発戦闘から、複数戦をつなぐ最小ループへ進む。

実装対象:

- ラン開始
- 戦闘選択
- 報酬選択
- 次戦への持ち越し
- 簡易ボス戦
- ラン終了

完了条件:

- 3戦から5戦程度の短いランを最後まで遊べる。
- 戦闘後の選択が次戦に影響する。
- 敗北またはクリア後に、すぐ次のランを開始できる。

## 常時守る運用

- 採用するルールは、実装前に仕様書へ明記する。
- 未確認ルールは推測で実装しない。
- 仕様変更をしたら、実装とテストを同じタイミングで更新する。
- 作業完了時は、検証を通してからコミットする。

## 次に着手する候補

直近では、Phase 1のうち次の順で進める。

1. CPU選択理由をログへ出す。
2. 対象選択中の盤面ハイライトを強化する。
3. 直近アクションのダメージ、回復、撃破表示を改善する。
4. CPUターン速度設定を追加する。
5. 手動プレイ20戦分の不具合メモを残す。
