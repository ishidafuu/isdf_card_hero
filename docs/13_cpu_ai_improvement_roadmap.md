# CPU AI改善ロードマップ

## 目的

`stable` を破綻検出用の基準AIとして維持しながら、`strong` を段階的に強化する。
強化は「勝率が上がったように見える」ではなく、同一seed比較、判断差分、回帰テストで説明できる変更だけを採用する。

最終的には、通常カード戦で `strong` が `stable` に対して安定して勝ち越し、かつ進行不能や極端な長期戦を増やさない状態を目指す。

## 現在地

実装済み:

- `stable` / `strong` AIプロファイル
- 陣営別AI指定
- カードヒーロー部.comの部員評価 `Pro黒` / `Pro白` の構造化取り込み
- マスター別の部員評価を使ったランダムデッキ重み付け
- `npm run validate:auto-play`
- `npm run benchmark:ai`
- `npm run diff:ai`
- `strong` の同一ターン内ビーム探索
- HPレース中の直接マスター打点評価補正
- `diff:ai` のターン範囲指定、盤面/手札/山札要約、初回/最大スコア差分サマリ
- 直打点がある局面での非撃破削り、ためる、移動/召喚探索ボーナスの局面限定抑制
- レベルアップ筋を守るシールド評価の補正
- 負け試合artifact出力

直近の確認結果:

- 補正前、seed `430` / `431` の4戦では `strong` が0勝。
- マスター打点評価補正後、seed `430` / `431` の4戦では `strong` が3勝。
- seed `430` から `434` の10戦では `stable` 5勝 / `strong` 5勝。
- seed `430` から `439` の20戦では `stable` 12勝 / `strong` 8勝、long game warning 1件。

残っている主な課題:

- `strong` がシールド、移動、ウェイクアップ、非撃破攻撃の長期価値に引っ張られる。
- 終盤で `focus` や非撃破の削りが多く、勝ち切り/負け回避の評価が弱い。
- `strong` の改善がseed範囲によって勝率差として安定していない。
- まだ相手ターンの浅い読みを「行動単位の比較」に直接使えていない。

## 基本方針

- `stable` は壊さない。基準AI、warning seedレビュー、通常100戦検証の既定として残す。
- `strong` だけを実験先にする。重い探索や局面別補正はまず `strong` に限定する。
- seed単位で改善/悪化を記録する。採用理由は「どのseedのどの判断が変わったか」まで追える状態にする。
- 勝率だけでなく、failure、warning、平均step、最大step、長期戦seedを同時に見る。
- 大きなminimaxへ飛ばない。まずは既存の合法手列挙、評価関数、浅い同一ターン探索の品質を上げる。
- 1コミット1テーマにする。ベンチ整備、評価補正、テスト追加、UI変更を無理に混ぜない。

## 共通検証ゲート

小変更の最低ゲート:

```sh
npm run build
npm test
git diff --check
```

AI評価変更の最低ゲート:

```sh
npm run benchmark:ai -- --seed-start 430 --count 5 --max-steps 700 --max-turns 160
npm run diff:ai -- --seed 430 --direction challenger-as-cpu --max-diffs 12
```

採用前の確認ゲート:

```sh
npm run benchmark:ai -- --seed-start 430 --count 10 --max-steps 700 --max-turns 160
```

節目の確認ゲート:

```sh
npm run benchmark:ai -- --seed-start 400 --count 50 --max-steps 700 --max-turns 160
npm run validate:auto-play -- --seed-start 400 --count 100 --max-steps 700 --max-turns 160
```

採用基準:

- failure 0。
- unresolved level-up 0。
- warningが増える場合は、該当seedのartifactレビューを必須にする。
- `strong` が対象seed範囲で明確に悪化する変更は戻す。
- 勝率が五分でも、既知の悪手が消え、warningや最大stepが悪化しないなら採用候補にできる。

## Phase A: 診断基盤を強くする

状態: 完了。`diff:ai` で `--turn-from` / `--turn-to` を指定でき、差分ごとにHP、石、手札/山札/捨て札、盤面要約、直近ログ、スコア差分を確認できる。

目的:

負けseedを見たときに、どの局面で `strong` が悪い選択へ分岐したかを短時間で追えるようにする。

実装候補:

- `diff:ai` に `--turn-from` / `--turn-to` を追加する。
- `diff:ai` にHP、盤面、手札枚数、デッキ残量の要約を1行で出す。
- `benchmark:ai` のartifactに、challenger負けだけでなくlong game warning試合も保存する。
- 負け試合の最後20手だけでなく、初回差分、最大スコア差分、勝敗直前差分を抽出する。

完了条件:

- 負けseedを1つ指定すると、初回分岐、終盤分岐、勝敗直前の判断差が読める。
- artifactを開かなくても、CLI出力だけで次に見る局面が分かる。

直近候補:

```sh
npm run diff:ai -- --seed 431 --direction challenger-as-cpu --max-diffs 16
npm run diff:ai -- --seed 432 --direction challenger-as-player --max-diffs 16
```

## Phase B: 直接勝ち筋と負け筋の評価

状態: 着手中。seed `431` / `432` のレビューを起点に、直接マスター打点がある局面で `strong` が非撃破削りやためるへ寄りすぎる症状を局面限定で抑制した。

目的:

マスターHPレース、リーサル、被リーサルを、盤面価値より優先して扱う。

実装候補:

- 直接マスター打点候補があるとき、非撃破のモンスター削り、ためる、移動、召喚の探索ボーナスを局面限定で抑える。
- 相手の次ターン最大マスター打点を、候補手ごとの詳細評価に直接入れる。
- 自分が次ターン倒される可能性がある場合、シールドや敵前衛除去の価値を上げる。
- 相手HPが3以下の時は、1点打点、2点打点、次ターン追加打点を別枠で加点する。

注意:

シールドを一律に強めると悪化したため、`致死圏の高価値味方`、`マスター被リーサル回避`、`次ターンレベルアップ維持` のように条件を絞る。

完了条件:

- seed `430` / `431` の改善を維持する。
- seed `430` から `439` の20戦で `strong` が五分以上、warning増加なし。
- 直接打点より非撃破削りを選ぶ差分が減る。

## Phase C: 防御とシールドの精度改善

状態: 着手中。次ターンのレベルアップ筋や致死圏の高価値味方を守るシールドを少し強めた。シールド過多を避けるため、低価値/非脅威対象への一律加点は行わない。

目的:

「守るべきカード」と「守らなくてよいカード」を分け、石を無駄に使わない。

実装候補:

- シールド価値を `守った後に次ターン何を得るか` で評価する。
- Lv2以上、高火力後衛、次ターン撃破/レベルアップ可能な前衛だけを強く守る。
- すでにマスターHPレースで負けている場合、盤面防御よりマスター打点や被リーサル回避を優先する。
- 低価値モンスターへの連続シールドを抑える。

完了条件:

- seed `431` の `challenger-as-cpu` で、必要なシールドを捨てる局面が減る。
- 逆にシールド過多で打点を逃す局面が増えない。

## Phase D: 移動/配置の長期価値改善

目的:

後衛を後列に置く、前衛を前列に出す判断を強くしつつ、移動だけでテンポを失わないようにする。

実装候補:

- 移動後の攻撃筋を「同ターン」だけでなく「次ターンに相手が除去できるか」込みで評価する。
- 後衛が前列に残っているリスクを強める。
- ただしHPレースで劣勢なら、配置改善より直接打点を優先する。
- 同一ターン内の移動/入れ替え後に攻撃できない場合は追加ペナルティを入れる。

完了条件:

- warning seedで移動多発が再発しない。
- `strong` の最大stepが増えない。
- seed `432` / `433` の負け差分で、移動の過大評価が減る。

## Phase E: 終盤評価

目的:

山札切れ、手札上限、残りHP、残り石を終盤の勝敗に近い形で評価する。

実装候補:

- 相手HPが低い時は、盤面除去より継続打点を優先する。
- 自分HPが低い時は、相手の最大打点を減らす行動を優先する。
- 山札残り2枚以下では、ドロー、リフレッシュ、長期戦化を別評価にする。
- 終盤の `focus` は、次ターンの具体的打点につながる時だけ高評価にする。

完了条件:

- long game warningが増えない。
- seed `437` のlong game warningをレビューし、進行不能ではなく終盤評価不足なら改善対象にする。

## Phase F: 相手ターンの浅い読み

目的:

候補手を選んだ後、相手の次ターンで最も痛い1手または短い連鎖を見て、明らかな自滅手を避ける。

実装候補:

- 相手の最大マスター打点を候補手ごとに評価する。
- 自軍の高価値モンスターが次に倒されるかを見る。
- 相手が撃破からレベルアップできるかを見る。
- まずは `strong` の詳細評価対象だけに限定する。

採用条件:

- 10seedベンチで処理時間が許容範囲。
- 50seedベンチでfailure 0。
- `stable` には入れない。

## Phase G: プロファイル整理

目的:

`strong` が十分に安定したら、強さと速度のバランス別にプロファイルを整理する。

候補:

- `stable`: 基準、軽量、検証既定。
- `strong`: 速度を保った強化AI。
- `analysis`: 重い探索や診断用。UI既定にはしない。

完了条件:

- 各プロファイルの用途がdocs、CLI、UIで一致する。
- `benchmark:ai` でプロファイルを入れ替えて比較できる。

## Phase H: マスター別カード評価の活用

目的:

攻略サイト上の `Pro黒` / `Pro白` 評価を、ルール実装ではなくメタ評価として扱い、マスター別のデッキ構成とAI判断に反映する。

現在の扱い:

- `memberRatings` としてカード定義に保存する。
- ホワイトマスターは `Pro白`、ブラックマスターは `Pro黒` を参照する。
- ランダムデッキ生成では、カテゴリ最低枚数と3枚制限を守ったまま、高評価カードが選ばれやすい重みにする。
- CPU AIでは、手札温存、召喚、特殊効果の手札選択に小さな補正として使う。

注意:

部員評価はゲームルールではなく、環境・構築知見の近似値として扱う。
そのため、個別局面のリーサル、被リーサル、配置、石効率より優先しない。

完了条件:

- Deck Setupのランダム生成が、選択中マスターに応じて `Pro黒` / `Pro白` の評価を使う。
- カード詳細とCard Listで、参照している評価値を確認できる。
- ブラック/ホワイト別の100戦検証でfailureが増えない。

## 採用しない方針

- 全候補で深いminimaxを回す。
- 勝率だけを見て理由の分からない補正を入れる。
- `stable` の既定挙動を不用意に変える。
- warningが増えた変更をartifactレビューなしで採用する。
- CPU AI改善とルール分割を同じコミットに混ぜる。
- 部員評価をルールやカード効果の正解値として扱う。
